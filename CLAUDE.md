## Project Overview

Full-stack Kanban application. Backend is Quarkus 3 (Java 21) with MSSQL + Flyway. Frontend is Angular 21 with Tailwind CSS. Authentication is delegated to a separate Keycloak-based auth service via OIDC.

## Commands

### Backend (`kanban-backend/`)
```bash
# Dev mode with live reload (requires local MSSQL + Keycloak)
./mvnw quarkus:dev

# Package
./mvnw package

# Run tests
./mvnw test

# Build Docker image
docker build -f src/main/docker/Dockerfile.jvm -t quarkus/kanban-backend-jvm .
```

### Frontend (`kanban-frontend/`)
```bash
npm start        # Dev server on :4200 (proxied to backend :8082, auth :8081)
npm run build    # Production build
npm test         # Vitest tests
```

## Architecture

### Backend Layers
1. **Resource** (`/resource`) — JAX-RS endpoints under `/v1/*`. Security enforced via `@RolesAllowed("user")`. Endpoints are stateless; user identity comes from the JWT `sub` claim mapped to `keycloakId`.
2. **Service** (`/service`) — Business logic with `@Transactional`. `StorageService` handles Azure Blob Storage for attachments.
3. **Entity** (`/entity`) — Panache Active Record pattern (`PanacheEntityBase`). Use `.persist()`, `.findById()`, and custom named queries directly on entities.
4. **DTO** (`/dto`) — Separate request/response DTOs to decouple API contracts from entities.

### Frontend Layers
- **Core** (`app/core/`) — `AuthService` (localStorage token), `AuthInterceptor` (injects Bearer token), `AuthGuard` (route protection), and all API services.
- **Features** (`app/features/`) — Standalone Angular components grouped by feature: `board`, `projects`, `auth`, `login`.
- **Shared** (`app/shared/`) — Reusable components (e.g., navbar).

Board drag-and-drop is handled by `@hello-pangea/dnd`. State is managed with RxJS services (no NgRx).

### Auth Flow
Quarkus OIDC validates JWTs issued by Keycloak (`realm/sprj`). The frontend sends credentials to a **separate** auth backend (`/v1/auth/login`, proxied to `:8081`). Tokens are stored in localStorage. The JWT `sub` claim is used as the `keycloakId` to identify users in all backend entities.

### Database
Flyway migrations run automatically on startup (`classpath:db/migration`). Schema is never auto-generated by Hibernate (`hibernate.hbm2ddl.auto` is not set). Six tables: `projects`, `project_members`, `kanban_columns`, `cards`, `attachments`, `notifications`.

## Key Configuration

### Backend (`application.properties`)
Required env vars at runtime:
| Variable | Purpose |
|---|---|
| `DB_URL` | MSSQL JDBC connection string |
| `DB_USERNAME` / `DB_PASSWORD` | Database credentials |
| `OIDC_AUTH_SERVER_URL` | Keycloak realm URL |
| `CORS_ORIGINS` | Allowed CORS origins |
| `AZURE_STORAGE_CONNECTION_STRING` | Azure Blob Storage |
| `AZURE_STORAGE_CONTAINER_NAME` | Blob container for attachments |

Dev defaults: MSSQL on `localhost:1433`, database `sprj_kanban`, Keycloak at `http://localhost:8080/realms/sprj`, Azurite for storage.

### Frontend (`proxy.conf.json`)
Dev proxy routes:
- `/v1/auth/**` → `http://localhost:8081` (auth backend)
- `/v1/**` → `http://localhost:8082` (this backend)

Production API base: `https://kanban-api.servulo.dev.br`

## Deployment

Azure Container Apps via Azure DevOps pipeline (`kanban-backend/azure-pipelines.yml`):
1. Maven build (tests skipped)
2. Docker image pushed to `sprjacr.azurecr.io`
3. Azure Container App `sprj-ca-kanban-backend` updated in resource group `sprj-rg`

Production Docker image uses `Dockerfile.jvm` (RedHat UBI9 + OpenJDK 25).
